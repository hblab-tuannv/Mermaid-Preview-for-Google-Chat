Load-bearing, source-verified fact for epic MAIN (ADR-MAIN-008). Re-check the version if `node_modules/mermaid/package.json` changes.

**Mermaid 11.15.0 renders flowchart labels as `<foreignObject>` HTML by default.** Verified in `node_modules/mermaid/dist/chunks/mermaid.core/chunk-CSCIHK7Q.mjs`: `getEffectiveHtmlLabels(config) = evaluate(config.htmlLabels ?? config.flowchart?.htmlLabels ?? true)` — it does **NOT** consult `securityLevel`. The extension's render config (`{startOnLoad:false, securityLevel:'strict'}`, ADR-MAIN-003) does not set `htmlLabels`, so it defaults to **true** ⇒ flowchart SVG contains foreignObject. `strict` only `removeScript`s the HTML, still emits foreignObject. (Common trap: assuming strict ⇒ htmlLabels:false. It does not in this version.)
**Why it matters:** an SVG with `<foreignObject>` loaded via `<img>` and drawn to `<canvas>` taints the canvas (`toBlob` throws SecurityError). So "PNG with crisp text for ALL diagram types via img→canvas" is NOT universally deliverable.
**EMPIRICALLY CONFIRMED in real Chrome (DevTools, scale 3x, transparent canvas, blob-URL→img→drawImage→toBlob):** flowchart = 7 foreignObject / 0 `<text>` → canvas TAINTED → `toBlob` throws `SecurityError: Tainted canvases may not be exported` (and `getImageData` throws). NOT fixable via config: even with effective `flowchart.htmlLabels=false` (verified via `getConfig`) AND a frontmatter `config:` block, Mermaid 11.15.0 STILL emits 7 foreignObject / 0 text. sequenceDiagram = 0 fo / 7 `<text>` → canvas NOT tainted → `toBlob` OK, 3x PNG crisp + transparent (screenshot-confirmed). So the taint is real and config-unfixable; native-text diagram types (sequence/class/state/er) export fine.
**How to apply:** SVG export (XMLSerializer of the clone, no canvas) is always correct/lossless for every diagram type — make it the **universal quality path**. PNG (canvas) is **best-effort**: correct for `<text>`-label types (sequence, class, …), but for flowchart its foreignObject labels taint the canvas so `toBlob` throws (always observable, never a silent blank — that's why a throw-keyed fallback is sufficient). Timeline `byFo` is gated by `textPlacement==='fo'` (not default) and wrapped in `<switch>`+`byTspan` fallback, so it degrades gracefully — flowchart htmlLabels is the real default risk, and flowchart is the most common type.

**Two ways to fix PNG-for-flowchart were BOTH rejected in ADR-MAIN-008 — and the second is the trap:**
- (a) set `htmlLabels:false` globally in US-003 render config → changes ON-SCREEN rendering of every shipped diagram (HTML vs SVG text wrap/style differ) ⇒ regression to shipped US-003..006 + edits accepted ADR-MAIN-003 ⇒ needs human sign-off, not an architect unilateral edit.
- (b) re-render `htmlLabels:false` "only for the throwaway PNG raster source" via the `MermaidRenderer` seam — **this is NOT isolated, despite looking like it.** Mermaid `initialize` is a GLOBAL SINGLETON, and `render.ts`'s `defaultRenderer` caches `initializedTheme` and SKIPS re-init on an unchanged theme. So a transient `initialize({htmlLabels:false})` leaks onto the singleton; the next on-screen render of the same theme hits the cache guard, skips re-init, and inherits `htmlLabels:false` → the exact US-003..006 regression, now transient and harder to spot. To do (b) safely you'd have to save/restore global htmlLabels AND bust the `initializedTheme` cache — not worth it.

**Decision recorded in ADR-MAIN-008 (option ii):** PNG rasterizes the ON-SCREEN clone as honest best-effort (no re-render, no global-state touch); SVG carries flowchart quality. Lesson: "re-render via the seam is isolated" is FALSE — global `initialize` + theme cache leak. When a Mermaid host-config override is needed only transiently, treat it as global mutation, not a local seam call.

**Gate outcome (RESOLVED — human gate 2026-06-12, Option A, CR-MAIN-2026-06-12-01):** the CONDITIONAL design gate fired (collides with Gate-1 locked "PNG crisp text") and the human DECIDED **Option A**: keep BOTH buttons; SVG = primary best-quality path; PNG = best-effort with **deterministic auto-fallback to SVG download + a light non-blocking notice** on `toBlob` SecurityError/null-blob; NO foreignObject→text rewrite, NO heavy renderer. Seam grew a third injectable `Notifier` (raster + save + notice) so the fallback (rasterizer rejects → assert SVG saved + notice fired) is jsdom-testable. ADR-MAIN-008 stays Accepted, now human-approved → Gate-3 GREEN. Pattern to reuse: when a best-effort capability can hard-fail for the common case, pair it with a deterministic fallback to the always-works path + a user notice, rather than silent no-op. See [[overlay-and-control-cleanup-patterns]] for the control-attach/cleanup half that WAS reusable as-is.
